## The vulnerability (context)

`query_csv()` in `src/agents/skill_csv_analyzer/tools.py` executes an LLM-authored expression via
`eval(expression, {"__builtins__": {}}, {"df": df, "pd": pd})`, guarded only by a regex blocklist
(`_BLOCKED_PATTERNS`). The blocklist is bypassable (e.g. `df.__getattribute__('__cla'+'ss__')`,
un-blocked `__reduce__`/`vars()`/`type()`, object-graph traversal to `os`/`subprocess`). Because
the expression is generated by the LLM after it reads untrusted CSV content, a malicious CSV can
drive indirect prompt injection into arbitrary code execution. `{"__builtins__": {}}` is not a
sandbox.

## What to build (recommended route)

Replace free-form `eval` with parameterized, allowlisted operations the agent selects by name —
no string is ever interpreted as code. Suggested tools:

- `describe_csv(csv_path)` — keep.
- `aggregate_csv(csv_path, group_by, metric, agg, filters)` — `agg ∈ {sum,mean,median,min,max,count,nunique,std,var}`.
- `value_counts_csv(csv_path, column, top_n)`
- `filter_count_csv(csv_path, filters)`
- `sort_head_csv(csv_path, sort_by, ascending, n, columns)`

Each handler validates column names against `df.columns`, validates `op`/`agg` against enums,
coerces filter values to column dtype (never interpolated into code), calls pandas directly, and
keeps the existing 50-row output cap. Also update `AGENT.md` (and `skill.yaml` if it enumerates
tools) so the agent uses the new operations.

Fallback only if free-form expressions are deemed essential: an `ast.parse(mode="eval")` allowlist
evaluator (no `eval`) — see the spec. Do **not** keep `eval`.

## Tests (add under `tests/`)

Security (must be blocked, zero side effects): `__class__/__mro__/__subclasses__` walk; the
`'__cla'+'ss__'` concat bypass; `pd.read_pickle` / `df.to_csv` IO attempts; `__import__`/`getattr`
attempts. Functional parity (must still work): group-by sum/mean, filtered count, value_counts
top-N, sort+head, describe; unknown column/op/agg returns a clean error; 50-row cap holds.
