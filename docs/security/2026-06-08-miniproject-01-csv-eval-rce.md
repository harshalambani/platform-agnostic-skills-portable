## Problem

`query_csv()` runs an LLM-authored expression through:

```python
result = eval(expression, {"__builtins__": {}}, {"df": df, "pd": pd})
```

The only guard is `_validate_expression()`, a regex blocklist (`_BLOCKED_PATTERNS`).
Regex blocklists do not contain a Python interpreter. Known bypasses that defeat this guard:

- **String construction** of blocked names: `df.__getattribute__('__cla' + 'ss__')` — the
  literal `__class__` never appears, so the regex misses it.
- **Un-blocked dunders/builtins**: `__getattribute__`, `__reduce__`, `__mro__`, `vars()`,
  `type(...)` are not in the blocklist.
- **Object-graph traversal** from `df`/`pd` to reach `object.__subclasses__()` and from there
  to `os`/`subprocess`-bearing classes — the classic `{}.__class__.__mro__[1].__subclasses__()`
  walk, reconstructed via the gaps above.

Because the expression is generated by the LLM, and the LLM has just read attacker-controlled
CSV content (`describe_csv` / row data), a malicious file can steer the model into emitting an
escape expression → **arbitrary code execution, arbitrary file read/write** in the user's
context. `pd` in scope widens the surface further (IO entry points).

`{"__builtins__": {}}` is **not** a sandbox; CPython has no supported in-process sandbox.

## Goal

Provide the same read-only analytical capability **without `eval`**, so that no crafted
expression — regardless of CSV content — can execute arbitrary code or touch the filesystem.

## Approach (recommended: structured operations, not free-form code)

Replace free-form expression evaluation with a small set of **parameterized, allowlisted
operations** the agent selects by name. The LLM picks an operation + typed arguments; Python
executes it with native pandas calls. No string is ever interpreted as code.

Concretely, redesign the tool surface to something like:

- `describe_csv(csv_path)` — unchanged.
- `aggregate_csv(csv_path, group_by: list[str] | None, metric: str, agg: str, filters: list[Filter] | None)`
  where `agg ∈ {sum, mean, median, min, max, count, nunique, std, var}` and each `Filter` is
  `{column, op ∈ {==,!=,>,>=,<,<=,in,notin,contains}, value}`.
- `value_counts_csv(csv_path, column, top_n: int)`
- `filter_count_csv(csv_path, filters: list[Filter])`
- `sort_head_csv(csv_path, sort_by, ascending: bool, n: int, columns: list[str] | None)`

Each handler:
1. Validates `column`/`group_by` names against `df.columns` (reject unknown → error string).
2. Validates `op`/`agg` against the enum.
3. Coerces `value` to the column dtype; never interpolates it into code.
4. Calls pandas directly (`df.groupby(...)[metric].agg(...)`, boolean-mask filtering, etc.).
5. Caps output rows (existing 50-row cap) and returns text.

This covers the real analytical use cases the current `query_csv` examples show
(`groupby(...).sum()`, `df[df[col] > x].shape[0]`, `value_counts()`, `describe()`) while making
RCE structurally impossible.

### If free-form expressions are deemed essential (fallback option)

Do **not** keep `eval`. Instead parse the expression with `ast.parse(expr, mode="eval")` and walk
the tree with an allowlist evaluator that permits only: `Expression`, `BinOp`, `Compare`,
`BoolOp`, `UnaryOp`, numeric/string/list/tuple literals, `Name` (only `df`/`pd`), `Subscript`,
and `Attribute`/`Call` **restricted to the existing `_ALLOWED_METHODS` set on `df`/Series only**.
Reject `Attribute` access to any dunder (`node.attr.startswith("__")`), reject all other names,
reject comprehensions/lambdas. This is more code and more error-prone than the structured
approach, so it is the fallback, not the default.

## Test plan

**Security (must all be blocked / return safe errors, never execute):**

1. `df.__class__.__mro__[1].__subclasses__()` and the full walk to a subprocess/os class.
2. `df.__getattribute__('__cla'+'ss__')` (string-concat bypass).
3. `pd.read_pickle('x')`, `pd.read_csv('/etc/passwd')`, `df.to_csv('/tmp/x')` (IO attempts).
4. `__import__('os').system('...')`, `getattr(df,'to_csv')` (builtin/getattr attempts).
5. Confirm no file is created and no command runs (assert via a temp dir watcher / monkeypatched
   `os.system`).

**Functional parity (must still work):**

6. group-by sum, mean; filtered row count; `value_counts` top-N; sort + head; describe.
7. Unknown column / bad op / bad agg → clean error string, no traceback leak.
8. 50-row output cap preserved.

**Regression:** existing csv_analyzer tests pass; agent end-to-end run produces a correct answer
on a sample CSV.

## Acceptance criteria

- No `eval`, `exec`, `compile`, or `runpy` anywhere in `skill_csv_analyzer`.
- All five security tests pass (operations rejected, zero side effects).
- All functional-parity tests pass.
- `AGENT.md` updated; agent selects the new tools without prompting confusion.
